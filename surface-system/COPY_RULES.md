# COPY_RULES

## Purpose

These rules constrain public-surface copy across all VERIFRAX hosts.

The objective is immediate classification, exact role clarity, and zero boundary collapse.

## Global rule

Every public page must answer, in order:

1. what surface is this
2. what does it do
3. what does it not do
4. what is the primary valid action here
5. what adjacent surfaces exist

## Required opening order

Every page must begin with:

1. surface identity
2. role statement
3. hard boundary statement

## Surface identity format

Format:

`VERIFRAX / <surface>`

Examples:

- `VERIFRAX / verify`
- `VERIFRAX / proof`
- `VERIFRAX / apply`
- `VERIFRAX / authority`

## Role statement rule

- one sentence only
- twelve words or fewer preferred
- no marketing language
- no metaphor
- no promise inflation
- no adjacent-role leakage

Examples:

- Deterministic verification of proof objects.
- Canonical proof publication surface.
- Applicant intake surface.
- Operational status surface.

## Hard boundary rule

Every page must state what it does not do.

Examples:

- This surface does not execute verification.
- This surface does not publish authority.
- This surface does not accept interpretation.
- This surface does not provide archive retrieval.

## Tone rules

Allowed:

- precise
- cold
- mechanical
- bounded
- declarative

Forbidden:

- inspirational copy
- urgency manipulation
- hero language
- exaggerated claims
- vague trust language
- anthropomorphic product copy

## Structure rules

Copy blocks must be one of:

- definition
- constraint
- input
- output
- action
- adjacency

Do not mix these in one paragraph unless strictly necessary.

## Role isolation rules

### www

May contain:

- root system statement
- commercial framing
- pricing
- legal pages
- links to public hosts

May not sound like:

- proof
- authority
- runtime
- archive
- verifier output
- status surface

### api

May contain:

- runtime identity
- health
- readiness
- version
- contract references

May not contain:

- pricing narrative
- proof publication language
- archive language

### proof

May contain:

- proof publication
- inspection
- retrieval
- certificate references

May not contain:

- verification execution language
- authority issuance language
- intake language

### verify

May contain:

- proof checking
- schema/spec references
- verifier UI wording

May not contain:

- proof publication wording
- archive wording
- commercial root wording

### apply

May contain:

- intake
- tasks
- submission
- confirmation

May not contain:

- jobs-board fluff
- docs role language
- proof/archive role language

### authority

May contain:

- current authority
- authority object
- digest
- verification note

May not contain:

- runtime role language
- proof role language
- intake role language

### runtime

May contain:

- runtime reference
- receipt model
- governed execution framing

May not contain:

- authority role language
- proof publication role language

### enforcement

May contain:

- rules
- boundary
- actions

May not contain:

- archive language
- authority language
- commercial root language

### archive

May contain:

- archive
- seal catalog
- historical retrieval

May not contain:

- live proof role language
- verification execution language

### docs

May contain:

- reference material
- canonical documentation entrypoints
- derivation notices where true

May not contain:

- product-homepage framing
- runtime claims unless sourced

### status

May contain:

- operational state
- incidents
- maintenance
- component status

May not contain:

- product narrative
- proof/archive/docs role language

## CTA rule

Each page gets exactly one primary action.

Adjacency links are not primary actions.

## Paragraph rule

Prefer short declarative blocks.

If a paragraph can be converted to:
- a label/value block
- a list
- a constraint line

do that instead.

## Final acceptance rule

A page fails copy review if a hostile reader cannot classify it in under three seconds.
